# 首轮样例共性规则

TLDR: 基于首批 5 份样例，当前已能先固定字段完整性、主线拆分、脱敏边界、内部资料归类和入库判断这 5 条共性规则，后续先按这些规则继续补样例，不再每份文件都从头判断。

## 一、字段规则
- 每份样例都必须先补最小元数据：原始文件名、原始文件格式、文档分类、内容主题标签、转换状态、版本信息。
- 再补该模板的核心字段，不强求一次补满所有细项。
- 缺失字段可以留空或标“待补”，但不能凭感觉补事实。

## 二、主线拆分规则
- 一份文件如果同时包含“正文内容”和“执行指令/补充要求/后续动作”，入库时优先保留正文主线。
- 执行指令、技能创建、MCP 搜索要求、继续补充说明，不直接当成知识正文。
- 如果后半明显切到另一个主题，应拆到对应页面或对应知识域，不整份硬塞进一个模板。

## 三、证据与事实规则
- 预算、联系人、政策依据、客户名称等字段，只有原文明确出现时才可写入。
- “需要去查”“待搜索”“后续补充”这类内容，只能记为线索，不能写成已确认事实。
- 联系人证据不足时，只保留单位级信息和公开接口/公开入口线索。

## 四、脱敏规则
- token、key、appSecret、账号口令、本地地址、部署命令、内部网关、Webhook 等内容不直接入知识库正文。
- 可保留“用途、能力、组件关系、公开接口地址”这类结构性信息。
- 若一份文件敏感内容占比过高，先标记为“不建议直接入主业务库”，必要时只保留脱敏摘要。

## 五、归类规则
- 产品/设备、方案/案例、政策/官方文件、单位/联系人、内部资料边界要分开处理。
- 内部分类口径、能力地图、平台理解框架，默认不进入主业务知识库；若要保留，优先放入内部资料库。
- 半官方结构稿、方案总纲、投标骨架可以入库，但必须显式标记“非正式文件”或“结构参考稿”。

## 六、是否适合直接入 Dify 的判断规则
- 适合直接入：主线清晰、事实边界明确、敏感信息可控、分类稳定的资料。
- 适合拆分后再入：正文和指令混杂、主副主题叠加、部分字段缺失但主线明确的资料。
- 不建议直接入：敏感配置过重、内部实现信息过多、主题耦合严重且暂未拆开的资料。

## 七、当前已验证的典型情况
- [[样例_长风知识库内容提取]]：主线明确，但后半需拆回商机评分副线。
- [[样例_贵州省林业无人机方案及报价]]：方案正文与执行指令混合，需先抽正文主线。
- [[样例_完整版面对政府]]：可入库，但必须标成结构参考稿，不能冒充正式政策。
- [[样例_重庆申请公开数据资料]]：联系人不足时只留单位级和接口级线索。
- [[样例_网站功能_内部边界]]：更适合内部资料库，不直接进主业务库。

## 八、按近期长风样本补出的结构化规则
- `D:\长风资料\长风` 这类目录下，产品参数卡、内部调研 PDF、跨厂家底表不能套同一模板；应先判断是“单机主档、调研线索、对比底表”中的哪一类。
- `热气球/` 这类特种载体资料，不直接套普通无人机产品模板；更适合先按“专题总页 → 型号底表 → 地方合规流程 → 报价证据”分层，再决定哪些字段可回填到产品层。
- `梦之鹰/` 这类单厂家资料包，不要先平铺多个碎页；更适合先定“厂家主档 → 多型号产品主档 → 配件/报价证据”三层关系。
- 价格、税运、标配/选配、当日有效这类字段默认视为强时效信息，只能挂证据层或报价层，不直接写成长期稳定事实。
- 图片型报价、截图型参数、内部整理版流程稿，默认先记证据类型和复核状态，不直接升格为正式主档事实；后续证据等级统一按 `[[16_证据链等级说明]]` 的 L1-L5 口径理解。
- 方案类资料至少还要再区分“综合方案总纲 / 客户汇报版 / 内部执行版”三种形态：总纲优先校准章节骨架与数据来源位，客户版优先校准价值表达与可公开边界，内部执行版优先校准 WBS、成本、岗位、收款与风险字段，不混成同一主档。
- 凡出现内部毛利率、岗位安排、详细成本、现金流、收款计划的方案材料，默认先挂内部执行模板或证据层，不直接进入对外方案主档正文。
- 凡出现“标题后需注明数据来源”的方案总纲，默认先标记为结构参考稿/待补证据状态，不直接当成已核实成稿。
- 凡是“完整方案必备内容”这类高密度提纲，优先把它理解成章节字典 + 附件清单 + 项目类型补充项，不直接当成某个真实项目方案正文。
- 凡是“流程图模块 / 技术实现载体 / 关键特性”这类资料，优先拆成架构设想、技术映射、代码骨架要求、敏感配置边界和待办，不直接并入方案正文或知识主档。
- 凡是“数据库必备内容”这类字段清单型资料，优先理解成辅助资料库注册表，不只看主档字段，还要补支撑库、测试池、模拟数据层与竞争对手资料库入口。
- 凡是“竞争对手分析”这类超长需求稿，优先拆成方法论、执行计划、功能注册/扩展基座、UI/交互、数据治理/预警、人工补录与置信度治理几层，不直接当单一方案文档。
- 凡是“独立子系统重构执行文档”这类资料，优先把目录隔离、执行文档包、状态文件、常驻 Agent 分工、功能注册表等治理要求单独抽出，避免与业务规则正文混写。
- 凡是“商机猎手 / 智能体团队 / 分省抓取说明”这类资料，优先拆成分省抓取范围、角色工作流、字段抽取与评分口径、推送规则、异常修正清单、脚本隔离要求几层，不直接并入知识库结构化主线。
- 凡是像“贵阳观山湖区公园方案”这种持续叠加追加要求的超混杂项目稿，默认要把主方案、生成约束、追加需求、附页要求、一期/二期边界分开，不把后补要求直接并入主方案事实层。

## 九、下一步怎么用
1. 后续新增样例先对照本页判断能否直接入库。
2. 若命中“拆分后再入”，先拆主线，再补对应模板，不直接先抽知识页。
3. 若命中“特种载体 / 单厂家报价包 / 对比底表 / 方案总纲 / 客户版 / 内部执行版”，优先按结构化模板反推字段和分层关系。
4. 若命中“内部资料边界”，优先判断是否只保留在内部资料库。
5. 若遇到新的混合型材料，再把新规则补回本页。
