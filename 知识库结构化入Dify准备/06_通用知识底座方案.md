# 通用知识底座方案

TLDR: 这页不是要替代当前主路线，而是解释这条线为什么会从“资料结构化入 Dify”继续升级成**通用知识底座**；执行上仍应先服从当前主路线：**阿里解析/OCR + MinerU + Qwen-Plus + 模板页/规则页**。

## 一、先说清这页的位置
这页属于**后续升级方向层**，不是当前样例执行的第一入口。

当前判断顺序应是：
1. 先确定主路线是否跑得稳
2. 再确定当前执行闭环怎么落
3. 最后再看这条线如何升级成通用底座

所以这页回答的重点不是“现在每一份文件立刻怎么处理”，而是：
**为什么当前主路线不应只停留在 Dify 入库，而应继续长成可复用的知识底座。**

## 二、当前主路线仍然是什么
当前更稳的主路线仍然是：
- 文档解析 / OCR：阿里工具链或同类稳定解析工具
- Markdown 中间层：MinerU 或同类方案
- 分类与字段抽取：Qwen-Plus
- 入库与分层判断：现有模板页 + 规则页

当前不建议这样理解：
- 不把 DeepSeek 当默认主抽取链路中心
- 不把 Logics-Parsing 直接当全库默认主路线

这条主路线先解决的是：
- 文档能不能稳定解析
- 能不能产出统一 Markdown / 结构化中间层
- 能不能稳定分类、抽字段、判模板
- 能不能按规则决定直接入、拆分后再入、暂不入

## 三、为什么这条线不能长期只停在“Dify 入库”
如果目标只是“文档进 Dify 就结束”，那很多后续需求会接不上。

你后续真正要支撑的，更像是：
- 方案生成系统取材
- 招标参数自动匹配
- 产品 / 厂家 / 检验报告联查
- 多来源资料统一归档
- 后续 API 或 Agent 工作流复用

所以更稳的理解是：
- **Dify 是当前重要消费端之一**
- 但长期目标不应只是“为 Dify 造一套流程”
- 更合理的是先沉淀底层知识结构，再决定由哪些系统来消费

## 四、从当前主路线自然长出来的三层底座

### 1. 原始文档层
作用：保留原件与来源，不改写事实。

最少保留：
- 原始文件名
- 原始路径
- 文件类型
- 文件哈希
- 来源目录
- 处理批次
- 处理状态

### 2. 标准化中间层
作用：把异构资料先变成统一可处理形态。

当前更适合承接：
- OCR 结果
- 文档解析结果
- Markdown 清洗稿
- 表格抽取结果
- 字段抽取 JSON
- 命名结果
- 置信度 / 错误记录

### 3. 知识主档层
作用：让后续系统按“知识对象”而不是按“原始文件”来使用内容。

当前至少应逐步固定 6 类主档：
- 产品主档
- 厂家主档
- 报告主档
- 政策主档
- 场景主档
- 方案主档

## 五、这页和当前执行层怎么衔接
这页不是另起一套口径，而是对当前执行层做上位解释。

### 当前执行层先做什么
先做：
- 样例验证
- 模板定口径
- 规则定边界
- 参数映射
- 去重
- 证据等级

### 为什么这些事值得先做
因为它们其实是在给后续底座打基础：
- 模板决定主档长什么样
- 规则决定哪些内容能升格为事实
- 参数映射决定后续能不能统一检索和匹配
- 去重决定主档是否会碎裂
- 证据等级决定后续治理能不能成立

## 六、当前更适合先固定的主档理解
这一步不是要求立刻大批量落所有主档，而是先统一“对象层”的理解。

### 1. 产品主档
关注：产品名、型号、类别、厂家、关键参数、关联报告、关联证据。

### 2. 厂家主档
关注：企业身份、核心产品、核心能力、资料包关系、关联案例与证据。

### 3. 报告主档
关注：报告编号、产品、型号、机构、日期、依据、结论、有效期。

### 4. 政策主档
关注：标题、发文单位、地区、时间、任务、适用层级、关联场景。

### 5. 场景主档
关注：场景名称、地区、行业、痛点、能力要求、关联政策、关联案例。

### 6. 方案主档
关注：方案名称、适用地区、适用场景、推荐设备、推荐厂家、预算锚点、证据来源。

## 七、从通用底座继续升级，会走到哪里
当主档、证据链、字段索引逐渐稳定后，这条线可以继续升级成更完整的治理底座。

后续可以补的治理能力包括：
- 生命周期状态
- 日期有效性
- 责任归属
- 替代关系
- 信源分级
- 审计记录
- 人工复核回流

但这里要注意：
**这是升级方向，不是当前样例阶段的第一优先表达。**

## 八、当前最稳的执行顺序
1. 先按当前主路线跑通解析、抽取、模板化、规则判断
2. 再用样例把模板、字段、判重、证据等级收稳
3. 再逐步形成 6 类主档
4. 再考虑 API、方案生成、参数匹配等多消费端复用
5. 最后再逐步补生命周期、权限、版本、审计等治理能力

## 九、这页的边界
这页不负责替代：
- `05_Dify工作流最小落地方案.md` 的最小工作流说明
- `07_四目录可执行实施方案.md` 的分目录执行安排

- `09_去重规则.md` 的具体判重规则
- `16_证据链等级说明.md` 的证据等级定义

这页只负责回答：
**为什么当前这条线值得按“可复用底座”来建设。**

## 十、当前一句话执行建议
先不要把这页当成当前操作手册；当前仍应优先跑稳 **阿里解析/OCR + MinerU + Qwen-Plus + 模板页/规则页** 的最小闭环，在这个基础上，再逐步把成果沉淀为可被 Dify、方案生成、参数匹配和 API 共同复用的通用知识底座。
