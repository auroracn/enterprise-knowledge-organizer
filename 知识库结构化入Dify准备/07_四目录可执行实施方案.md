# 四目录可执行实施方案

TLDR: 这页负责把当前主路线拆成 4 个执行目录去推进；阅读时也要分三层：**当前闭环怎么跑**、**扩展执行怎么铺开**、**远期治理怎么承接**。当前优先级仍是先跑通最小闭环，不是一次把四条线全部打满。

## 一、这页的定位
这页属于**当前执行方案层**。

它不负责重新定义主路线，而是负责回答：
- 四个目录分别承接什么
- 当前阶段先跑哪些样本
- 先把哪些对象和规则收稳
- 后续再怎么扩展

所以执行顺序上，这页应服从：
- `05_Dify工作流最小落地方案.md`
- `04_首轮样例共性规则.md`

## 二、先分清三层

### 1. 当前闭环
当前最优先的是：
- 让解析、抽取、规则判断、模板化这条链路稳定
- 让少量高价值样本能稳定落地
- 让模板、字段、判重、证据链先有统一口径

### 2. 扩展执行
在最小闭环稳定后，再扩大到：
- 多目录并行推进
- 多类样本覆盖
- 6 类主档逐步成形
- 正式主档页逐步落仓库

### 3. 远期治理
当目录执行成熟后，再继续承接：
- 生命周期治理
- 多消费端复用
- API / 方案生成 / 参数匹配联动
- 更稳定的主档与证据治理机制

## 三、四个执行目录的角色分工

### 1. `D:\新维度测试资料`
定位：**产品、厂家、检验报告、参数表、报价表底座**

当前主要服务：
- 产品主档
- 厂家主档
- 报告主档
- 参数标准化
- 证据挂接

### 2. `D:\低空经济相关资料汇总（李伟）整理2025-05-23至2026-03-31 - 副本`
定位：**政策、场景、案例、区域机会底座**

当前主要服务：
- 政策主档
- 场景主档
- 案例主档
- 区域机会理解

### 3. `D:\长风资料 - 副本`
定位：**方案样本、厂家资料、产品手册、报价与商务底座**

当前主要服务：
- 方案主档
- 厂家主档
- 产品主档
- 报价证据层

### 4. `D:\aurora\Aurora知识库\Aurora知识库`
定位：**正式落地层**

当前主要服务：
- 承接正式主档页
- 承接规则、模板和说明页
- 承接索引、日志和最终归档

## 四、当前阶段的统一执行原则
- 不把“全文转 Markdown”当最终目标。
- 当前先以“主档实体 + 证据文档 + 结构化字段索引”为方向。
- Excel / 表格优先字段化，不强行走长文 Markdown 流程。
- 自动抽取结果不直接等于最终事实，必须保留证据路径和状态。
- 同一对象优先收敛到一个正式主档，不先铺碎页。
- 当前先跑少量高价值样本，不急着全目录一键打通。

## 五、当前闭环：先怎么跑

### 第一步：先跑跨目录最小样本组合
当前更适合先挑一组能覆盖主要类型的样本，而不是四个目录同时全面铺开。

建议至少覆盖：
- 新维度中的参数表、产品说明书、检验报告
- 低空资料中的政策、场景清单、典型案例
- 长风资料中的方案样本、产品手册、报价样本

目标不是求多，而是先验证：
- OCR / 解析分流是否稳定
- 模板判断是否稳定
- 字段抽取是否稳定
- 规则判断是否稳定

### 第二步：先固化 6 类主档理解
当前至少先统一这 6 类主档：
- 产品
- 厂家
- 报告
- 政策
- 场景
- 方案

### 第三步：先把三类基础规则收稳
优先收稳：
- 参数映射字典
- 去重规则
- 证据链等级

### 第四步：再落正式主档页
当样本口径稳定后，再把高频对象正式落进 Wiki。

## 六、扩展执行：四条线分别怎么铺

### 1. 新维度线
当前目标：先跑通“产品主档 + 报告主档 + 厂家主档”的最小闭环。

优先样本类型：
- Excel 挂网参数表
- 产品说明书
- 检验 / 检定 PDF
- 3C / 合规类材料
- 同厂家多型号资料包

当前重点不是把每一份资料都立刻拆完，而是先把：
- 产品字段口径
- 报告字段口径
- 厂家归属判断
- 同对象去重规则

收稳。

### 2. 低空资料汇总线
当前目标：先跑通“政策主档 + 场景主档 + 案例主档”的最小闭环。

优先样本类型：
- 国家 / 省 / 市级政策
- 场景机会清单
- 场景能力清单
- 典型案例
- 项目方案文本

当前重点是先把：
- 政策任务拆条
- 场景能力归纳
- 案例与政策、场景的关联关系

收稳。

### 3. 长风资料副本线
当前目标：先跑通“方案主档 + 厂家主档 + 产品主档 + 报价证据”的最小闭环。

优先样本类型：
- 平台级方案
- 行业解决方案
- 单厂家资料包
- 产品手册
- 报价 PDF / Excel
- 截图型和图片型资料

当前重点是先把：
- 方案层、厂家层、产品层、报价证据层的边界
- 单厂家资料包的父子关系
- 价格有效期和证据状态

收稳。

### 4. Wiki 落地线
当前目标：用最少新增页面，把前面三条线稳定下来的对象正式落进仓库。

当前更适合先落：
- 总览页
- 主档目录页
- 少量高频主档页
- 参数映射字典
- 去重规则
- 证据链说明

当前不建议：
- 一开始给每个对象都建碎页
- 把原始全文都塞进主档正文
- 把实验产物和正式主档混写

## 七、远期治理：这页后续怎么承接 06
当四条线逐步稳定后，这页可以继续承接 `06_通用知识底座方案.md` 提到的升级方向，包括：
- 生命周期状态
- 日期有效性
- 主档替代关系
- 信源分级
- 权限、审计、版本
- Dify、API、方案生成、参数匹配等多消费端接入

但当前要注意：
**这部分属于远期治理，不应反压当前样本执行节奏。**

## 八、当前完成判断
如果以下几点成立，就说明这页对应的执行层已经基本跑顺：
- 四个目录的角色边界是清楚的
- 至少一批代表样本已经跑通“原件 → 中间层 → 主档/证据层”
- 6 类主档理解已经稳定
- 参数映射、去重、证据等级已有统一口径
- 高价值对象已经能开始正式落 Wiki

## 九、当前一句话执行建议
先把四个目录理解成**四条分工明确的执行线**，但不要一开始就四线全量铺开；当前更稳的做法仍是先用代表样本跑通最小闭环，再逐步扩成多目录、多主档、多消费端共用的知识底座。
