# 开发环境与实施指引

TLDR: 这页把“长风知识库结构化入 Dify准备”当前已沉淀的目录、技术栈、开发流程、Agent 分工、MCP 工具、环境搭建与执行顺序收成一份完整开发文档，后续默认先看这页再开工。

## 一、这份文档解决什么问题
- 给后续实际开工的人一个统一入口。
- 把“看哪些页、按什么顺序做、用哪些模型和工具、哪些内容不能直接入库”一次说清。
- 避免每次重新从散落的说明页、样例页、规则页里自己拼上下文。

## 二、当前项目的真实定位
这条线当前不应只理解为“把文件丢进 Dify”。

更准确的理解是：
- 先把多源资料整理成 **主档实体 + 证据文档 + 结构化字段索引** 的通用知识底座。
- 再让 Dify、方案生成系统、参数匹配系统、API、企业消息入口去消费这套底座。

所以 Dify 是当前主要消费端之一，但不是唯一目标本身。

## 四、当前推荐技术栈

### 1. 主模型路线
当前默认主路线：
- **Qwen-Plus**：主分类、字段抽取、模板判断、摘要生成
- **DeepSeek**：辅助润色、补充改写、非核心复核，不再作为默认主抽取链路

### 2. 文档处理路线
- **阿里 OCR / 文档解析工具**：扫描件、图片、PDF、Office 解析主链路
- **MinerU 或同类方案**：Markdown 中间层、清洗稿产出
- **规则页 + 模板页**：控制是否直接入库、是否拆分、应该落哪一层

### 3. 知识组织与存储形态
- **Markdown**：人工可读主载体、主档页、规则页、样例页
- **JSON / 字段抽取结果**：中间层结构化结果
- **Dify 知识库**：当前主要消费端
- **结构化字段索引 / 主档关系**：后续 API、方案生成、参数匹配共用底座

### 4. 推荐主档 schema
当前至少固定 6 类主档：
- 产品主档
- 厂家主档
- 报告主档
- 政策主档
- 场景主档
- 方案主档

### 5. 证据与治理能力
必须配套：
- 参数映射字典
- 去重规则
- 证据链等级 L1-L5
- 日期有效性
- 生命周期治理
- 人工复核与回流

## 六、标准开发流程

### 阶段 1：定模板
目标：同类文件换人处理，结果也基本一致。

要做：
- 固定模板类型
- 固定最小必填字段
- 固定命名规则
- 固定敏感边界
- 固定跳过边界

### 阶段 2：跑样例
目标：先用少量代表样本把口径跑稳。

要做：
- 从 `03_样例验收清单.md` 挑样本
- 先跑 5+1 已验证样例
- 对照模板补结构化结果
- 对照规则页做入库判断
- 把新增发现回写规则页

### 阶段 3：定规则
目标：把样例反馈沉淀成批量执行 SOP。

要做：
- 固化参数字段映射
- 固化判重主键
- 固化证据等级
- 固化拆分与跳过逻辑
- 固化人工复核点

### 阶段 4：批量执行
目标：按样本类型成批处理，不再每份临时判断。

要做：
- 分批处理目录资料
- 形成已入 / 待修 / 跳过清单
- 记录异常原因
- 每轮回写日志和检查点

## 八、Agent 分工建议
这条线最适合按“探索 → 规划 → 执行 → 复核”四段分工。

### 1. Explore / 探索 Agent
适合做：
- 快速扫目录和样本分布
- 查重名文件、样本形态、现有页面
- 确认哪些资料属于哪个模板

输入建议：
- 指定目录
- 指定想确认的对象类型
- 指定输出要简洁列清单还是要带判断

输出结果：
- 样本清单
- 文件形态判断
- 候选模板归类

### 2. Plan / 规划 Agent
适合做：
- 设计样例推进顺序
- 设计 schema 和字段
- 设计批量执行节奏
- 设计目录分工和落地页结构

输出结果：
- 分阶段计划
- 页面结构
- 字段设计
- 风险与边界说明

### 3. Executor / 执行 Agent
适合做：
- 逐份样本抽取
- 按模板整理 Markdown
- 回写样例页和规则页
- 更新索引与日志

输出结果：
- 样例页
- 模板页更新
- 规则页更新
- 索引与日志留痕

### 4. Verifier / Reviewer / Critic
适合做：
- 检查是否误把线索写成事实
- 检查是否遗漏敏感边界
- 检查是否把内部资料混入主业务库
- 检查字段是否统一、链接是否齐

输出结果：
- 审核意见
- 风险清单
- 缺漏清单

## 九、建议的 Agent 使用步骤

### 场景 A：处理一批新资料
1. 先用 Explore 扫目录和样本形态
2. 再用 Plan 确定先跑哪 3-5 个代表样本
3. 再用 Executor 按模板落样例
4. 再用 Verifier 复核事实边界、敏感边界、去重和证据等级
5. 最后回写 `[[更新日志]]` 和 `[[整理检查点]]`

### 场景 B：补一条新规则
1. 先定位该规则来自哪份样例或哪批目录
2. 用 Executor 更新对应规则页
3. 用 Reviewer 检查是否影响旧样例
4. 必要时同步修模板页和参数字典

### 场景 C：要接 Dify 或自动化流程
1. 先看 `[[05_Dify工作流最小落地方案]]`
2. 明确输入文件类型与输出字段
3. 明确哪些节点走 OCR、哪些走解析
4. 明确 JSON 输出字段
5. 明确直接入 / 待审核 / 跳过三种出口

## 十、MCP 工具使用建议
下面按这条线的实际需要给出建议，不是所有工具都必须上。

### 1. 文档与外部资料查询
- `context7`：查 Dify、SDK、框架、API 最新官方文档
- `Brave Search / WebSearch / jina search`：查公开政策、工具方案、模型能力、外部参考

适合场景：
- Dify API、工作流节点、知识库接入文档核实
- 阿里 OCR / 文档解析相关文档核实
- Qwen 模型调用参数和限制核实

### 2. 浏览器自动化类
- `browser_navigate`
- `browser_snapshot`
- `browser_click`
- `browser_fill_form`
- `browser_network_requests`

适合场景：
- 实际查看 Dify 控制台
- 核对工作流节点配置
- 验证上传、入库、字段映射页面
- 抓实际请求结构做对照

### 3. 代码与结构辅助类
- `lsp_diagnostics`
- `lsp_document_symbols`
- `ast_grep_search`
- `python_repl`

适合场景：
- 如果后续这条线出现脚本、清洗器、抽取器实现
- 检查 schema、脚本、字段映射逻辑
- 快速做小规模字段验证和数据对比

### 4. 记忆 / 项目上下文类
- `project_memory_read`
- `state_read`
- `notepad_read`

适合场景：
- 快速恢复当前项目约束
- 恢复长风主线和四页协作上下文
- 恢复规则收敛历史

### 5. 当前不建议滥用的方向
- 不要一开始就把所有 MCP 都串成超大自动化链。
- 不要没固定字段和规则前就直接上全量浏览器自动提交或全目录自动入库。
- 不要用自动化结果替代人工复核。

## 十二、字段与治理的最低要求

### 1. 最小元数据必填
- 原始文件名
- 原始文件格式
- 文档分类
- 推荐模板
- 主体名称 / 产品名称 / 单位名称
- 内容主题标签
- 转换状态
- 版本信息
- 原始路径

### 2. 必须统一的治理能力
- 字段映射统一看 `[[08_参数映射字典]]`
- 去重统一看 `[[09_去重规则]]`
- 证据等级统一看 `[[16_证据链等级说明]]`
- 敏感边界统一看 `[[04_首轮样例共性规则]]`

### 3. 默认高风险内容
以下内容默认不直接入主业务知识库正文：
- token / key / secret
- Webhook / 内网地址 / 本地服务地址
- 部署命令
- 强时效价格
- 内部毛利率、现金流、岗位安排
- 图片型报价、截图型参数、内部整理版流程稿

## 十三、当前最适合先做的真实动作
1. 继续按 `03_样例验收清单.md` 顺序补样例
2. 优先拿大文件和高混杂真实文件校准模板
3. 把 6 类主档 schema 稳住
4. 把参数映射、去重、证据等级做成硬规则
5. 再去做 Dify 节点配置和自动化批处理

## 十四、完成判断
如果出现以下状态，可以认为这条线进入稳定开发阶段：
- 同类文件基本能稳定选对模板
- OCR / 解析分流判断稳定
- 直接入 / 待审核 / 跳过三种出口稳定
- 6 类主档字段基本固定
- 样例页质量接近主档页要求
- 新增样本只需少量补规则，而不是每次重做方法论

## 十五、当前一句话建议
先按 **Qwen-Plus + 阿里解析/OCR + 规则页 + 模板页 + 样例先行** 跑稳最小闭环，再逐步往 Dify 自动化和企业级知识治理扩展，不要一开始追求全目录全自动。
