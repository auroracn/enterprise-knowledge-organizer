# Qwen-Plus + 阿里工具链方案

TLDR: 对长风当前“资料结构化入 Dify”这条线，更建议优先走 **Qwen-Plus（你当前阿里环境中的 3.6 版本）+ 阿里文档解析工具链 + Dify 工作流**，原因是国内可用性、接入稳定性、OCR/文档解析配套、后续维护成本都更合适；DeepSeek 可保留为备选，不建议作为当前主方案。

## 一、为什么当前更建议从 DeepSeek 切到 Qwen-Plus

### 1. 更贴近你现在的实际使用环境
- 你这条线不是单纯聊天，而是 **PDF / Word / Excel / 图片 → 解析 → 抽取 → 结构化 → 入 Dify** 的完整链路。
- 在国内环境下，Qwen 配合阿里生态工具更容易统一接入、统一维护，也更适合后续在 Dify 工作流里持续调用。

### 2. 这条线更需要“文档处理能力”，不只是大模型回答能力
- 当前难点不是单纯总结，而是：
  - 扫描件 OCR
  - 表格 / 参数 /章节结构识别
  - 按模板抽字段
  - 输出稳定 JSON / Markdown
  - 适配后续 Dify 入库
- 所以主方案应优先看“解析 + 抽取 + 工作流接入”，而不是只看单个模型的通用能力。

### 3. 更符合当前准备目录已经形成的口径
结合当前已跑通的样例和规则页，长风这条线已经更明确需要：
- 主线和执行指令拆分
- 敏感字段脱敏
- 联系人证据不足时只保留单位级信息
- 产品、方案、政策、联系人、内部资料分模板处理

这意味着模型方案必须支持：
- 稳定解析复杂文档
- 按模板做结构化抽取
- 配合工作流节点做分流处理

Qwen + 阿里工具链更适合做这件事。

## 二、建议的主方案结构

### 主模型
- **Qwen-Plus（以你当前阿里可用的 3.6 版本为准）**
- 主要职责：
  - 模板判断
  - 字段抽取
  - 摘要生成
  - 分类与命名建议
  - 敏感信息判定辅助

### 文档解析层
优先考虑阿里系文档解析 / OCR 工具：
- OCR：处理扫描件、截图、图片 PDF
- 文档解析：处理 Word / PDF / Office 文档正文、章节、表格
- 输出目标：尽量先转成 **Markdown / HTML / 可抽取文本块**

### 抽取层
- 用 Qwen-Plus 按模板抽取字段
- 输出统一 JSON
- 再把 JSON 转成模板化 Markdown

### 工作流层
- 用 Dify 工作流串起来：
  1. 上传或读取文件
  2. 预判文件类型
  3. 分流到 OCR / 文档解析
  4. 调 Qwen 做分类与字段抽取
  5. 根据模板生成标准 Markdown
  6. 进入 Dify 知识库或待审核区

## 三、建议的处理分流

### 1. 扫描件 / 图片类
- 先走 OCR
- 再进入 Qwen 抽取
- 如果文字极少、纯照片、无知识价值，直接跳过

这和我们当前规则页是一致的：
- 纯照片跳过
- 证据不足不强补

### 2. 纯文本 / Word / 规则清晰的 PDF
- 先走文档解析
- 拿到结构化文本后直接进入字段抽取
- 这类资料最适合首批批量化

### 3. 混合型资料
比如你现在已经遇到的：
- 方案正文 + 后续补充指令
- 单位线索 + 接口说明 + 部署命令
- 内部口径 + 对外资料混写

这类资料不要直接整篇入库，而是：
- 先让模型判断主主题
- 再按规则拆分
- 只保留主线正文进知识库
- 其余内容打到“内部参考 / 待整理 / 不入库”

## 四、对 Dify 入库最有用的输出格式

当前不建议模型最后只吐一段长摘要，建议固定成两层输出：

### 第一层：结构化 JSON
至少包含：
- 文档分类
- 原始文件名
- 主体名称 / 单位名称 / 产品名称
- 主题标签
- 核心摘要
- 关键字段
- 敏感边界判断
- 是否适合直接入 Dify
- 是否需要拆分
- 推荐模板类型

### 第二层：模板化 Markdown
按你已经做好的模板输出：
- 产品/设备
- 方案/案例
- 合同/商务
- 供应商/企业
- 政策/官方文件
- 单位/联系人

这样后续就能直接接你现在准备目录里的模板体系。

## 五、为什么 DeepSeek 不建议继续做主方案
不是说不能用，而是当前这条线里它更像备选：

### 可保留的地方
- 做补充摘要
- 做二次改写
- 做风格润色
- 做非核心抽取复核

### 不适合做主方案的原因
- 你当前场景更依赖文档解析和结构化工具链，不只是模型本体。
- 当前你后续还要接 Dify、模板、规则页、样例页，主方案更需要统一生态。
- 如果主方案本身不稳定，后面批量入库阶段会增加很多维护成本。

所以更合理的定位是：
- **Qwen-Plus + 阿里工具链 = 主方案**
- **DeepSeek = 备选或辅助模型**
