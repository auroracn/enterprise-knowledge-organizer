## 一、用户需求与使用场景

### 1.1 下游应用(不止一个)

当前知识库要同时服务**至少三个下游场景**,现有 Dify 机器人只是其中一个环节:

| # | 应用 | 用途 | 目前状态 |
|---|---|---|---|
| 1 | **知识库问答机器人** | 员工标书书写过程中"查我们有什么资料" | 已搭建(`长风知识库机器人.yml`) |
| 2 | **方案生成** | 输入主题/客户,产出解决方案 | 规划中 |
| 3 | **招投标书生成** | 输入标书要求,产出响应文件 | 规划中 |

### 1.2 对 KB 内容的引用预期

**方案生成**会同时引用:
- 最新**低空行业新闻**(公众号/新闻日报)
- 最新**无人机型号**(长风产品库 + 自动--产品库)
- **厂家技术方案**(合作供应商资料)
- **政策/行业背景**(政策文件 / 政府动态 / 林业知识)
- **话术/对客表达**(长风话术库)

**招投标书生成**会引用:
- **价格库**的参数偏离表模板
- **产品规格参数**
- **历史投标响应**
- **公司资质/供应商资质**

### 1.3 用户核心顾虑

> "目前只是起步阶段,新闻公众号也只是几天的数据,很可能以后越来越多。"

→ **新闻类数据的持续增长**会不会拖垮所有下游场景的检索质量?

### 2.2 当前 Dify 问答机器人的问题(`长风知识库机器人.yml`)

1. **参数提取器节点冗余**:只定义 `answer:string` 后原样透传,增加一次 LLM 调用和 JSON 解析失败风险,**应删除**。

## 三、架构建议

### 3.1 知识三层模型(按信号密度 + 时效性分)

> 关键原则:**不按内容类型分,按在检索里的角色分**。

| 层 | 成员库 | 特征 | 在检索里的角色 |
|---|---|---|---|
| **A 核心业务层** | 长风产品库 / 长风案例库 / 长风话术库 / 合作供应商 / 价格库 / 政府部门联系方式 / 长风内部资料 | 慢变、高相关度、直接进入标书/方案 | **默认主检索**,top_k 多给 |
| **B 结构化参考层** | 政策文件 / 官方指导 / 政府动态 / 贵州林业相关知识 / 自动--产品库 / 测试 | 中频更新、场景性调用 | **按主题命中时检索**,top_k 中等 |
| **C 资讯流层** | 自动--公众号 / 自动--新闻日报 / 自动--新闻条目 | 高频增长、单篇信息密度低、时效强 | **独立检索通道**,永不混入 A/B |

### 3.2 下游应用分拆(一场景一个 Dify App)

#### App 1 · 知识库问答机器人(当前)
- 挂载:A 层 + B 层(**不挂 C 层**)
- 新闻类问题由入口分类节点跳转 App 4

#### App 2 · 方案生成 workflow(**分路并行检索**,最后合流)
```
              ┌── 检索 A: 产品库+案例库+供应商         (主素材)
输入主题 ──┼── 检索 B: 政策+行业知识+政府动态        (合规/背景)
              ├── 检索 C: 最近 30 天新闻(时间过滤)    (时效引用)
              └── 检索 D: 话术库                        (文风)
                                ↓
                            LLM 合成方案
```
每路独立 top_k(A=10 / B=5 / C=3 / D=2),互不挤占。

#### App 3 · 招投标书生成 workflow
- 挂载:价格库 + 长风产品库 + 合作供应商 + 官方指导 + 长风案例库
- **不挂新闻类**
- 多一步"参数提取":从输入标书抽取技术指标,再用指标去检索价格库

#### App 4(可选)· 行业情报机器人
- 仅挂 C 层,专门答"最近低空行业动态"

### 3.3 新闻数据长期治理(应对增长)

#### 措施 1:**文档级时间元数据 + 检索时过滤**
- 每份新闻入库时加 metadata `发布日期`(你数据里已有字段,只是没写入 Dify metadata)
- Dify 检索节点开启 `metadata_filtering_mode`,过滤 `发布日期 >= 当前-30天`
- 无论累计多少,检索只在近 30 天池子里跑

#### 措施 2:**数据金字塔**
- 公众号原文:最近 **7 天**(过期归档/删除)
- 新闻日报:保留 **90 天**
- 月度摘要:每月 1 篇,长期保留

#### 措施 3:**热冷分库**
- `新闻--热点(近30天)` 和 `新闻--归档`
- 默认只挂热点库
- 定时脚本每日把 30 天前的文档从热点库迁到归档库

## 七、两个下游系统 Dify 接入审计(2026-04-24)

> 读代码结论,**不复述实现细节**,只记影响架构决策的关键点。
> 这些事实后续可能随系统升级变化,但**架构判断和契约**应保持稳定。

### 7.1 智能标书系统(`D:\changfeng\智能标书\yibiao`)

| 维度 | 现状 | 评价 |
|---|---|---|
| API 端点 | `/datasets/{id}/retrieve`(官方 Dataset API) | ✅ 生产级,稳定 |
| 抽象层 | `KBRetriever` ABC + `DifyKBRetriever` 实现 | ✅ 换后端不伤业务 |
| 编排 | `retrieval_orchestrator`:P1 本地政策 + P2 Dify **并列展示**不自动合并 | ✅ 设计克制,防幻觉 |
| 稳定性 | retry / health_check / SSL 灵活 | ✅ |
| Dataset 选择 | 单库(`knowledge_default_dataset_id`),legacy 支持多 id 逗号分隔 | ⚠️ 单库模式,将来 catalog 化后需重构 |
| 写入 | 有 `KnowledgeMaterialService` 双向同步 | ⚠️ 写入如果继续开放,易造成 chunking 不一致(见 §8.3 原则) |
| 入口 | `get_kb_retriever()` 单例 | ✅ |

**结论**:招投标系统 是**合格的参考实现**,抽象层和编排模式应被作为"共享 kb_client"的蓝本。

### 7.2 方案生成系统(`D:\aurora\方案生成系统`)

| 维度 | 现状 | 风险 |
|---|---|---|
| API 端点 | `/datasets/{id}/hit-testing` | ❌ **调试接口**,Dify 不保证稳定,升级时易坏 |
| 抽象层 | 无,`DifyKnowledgeClient` 直接 httpx | ❌ 迁移成本高 |
| Dataset 映射 | 硬编码 4 个:case / script / product / competitor_library | ⚠️ 账户实际 18 个库,**14 个闲置** |
| `search_all` | 并发打账户下**所有库**,按 score 排取 top_k×2 | ❌ 散弹检索:30 agent × 18 库 = 单次用户请求最多 540 次 Dify 调用 |
| Score 可比性 | 不同库返回的 score **不可比**(reranking 关闭),排序实际按 chunk 长度+噪声 | ❌ 新闻库高密度稀释业务库 |
| api_key | 硬编码在 fallback 分支 | ❌ 配置污染,无法分系统追踪 |
| Chunking 策略 | 上传时 `separator:"###"` + `max_tokens:500` | ⚠️ 与长风知识整理系统入库链路不一致,同一文档双源切法不同 |
| Agent 调用 | 30+ agent 通过 `base_agent.dify` 共享 client,部分用 `set_dataset(type)`,部分用 `search_all` | ⚠️ 无统一路由,scaling 隐患 |

**结论**:方案生成系统 现状在**起步阶段能跑**,但**规模上来必炸**。`search_all` 是最紧迫的隐患。

### 7.3 共同问题(两个系统的横向问题)

1. **API 端点不统一**(retrieve vs hit-testing)→ Dify 升级时 方案生成系统 先坏
2. **同一把 api_key**(`dataset-xxxxxxxxxxxxxxxxxxxxxxxxxx`)两边共用,无法分系统计费/限流/审计
3. **Metadata filter 全部未启用**→ 无法按日期/行业/厂商精准过滤
4. **无中央 dataset 目录**→ 库结构变化下游全要改代码

## 八、统一架构建议(目标:迁移/升级时框架不大改)

> 核心思想:**把"事实源"收敛到长风知识整理系统**,下游系统只读契约,不碰实现。

### 8.1 三份合约文件(单一事实源,放在长风知识整理系统)

#### `kb_catalog.yaml` — 库目录
```yaml
datasets:
  - id: 9b910bf7-...
    name: 长风产品库
    tier: A              # A=核心业务 / B=结构化参考 / C=资讯流
    roles: [product, 产品推荐, 技术参数]
    metadata_schema: product_v1
    ingest_source: manual   # manual=人工确认 / auto_news=新闻抓取
    retention_days: null    # null=永久, 30=只保留近30天
```
- **ingest_source 规则**(用户 2026-04-24 澄清):
  - `auto_news`:所有 `自动--` 开头的库(`自动--公众号` / `自动--新闻日报` / `自动--新闻条目` / `自动--产品库` / `自动--新闻条目`)
  - `manual`:其他所有库,人工确认内容

#### `kb_metadata_schema.yaml` — 通用字段合约
```yaml
common_fields:
  - name: 行业
    type: enum
    values: [林业, 消防, 应急, 通用航空, ...]
  - name: 文档类型
    type: enum
    values: [产品规格, 案例, 政策, 新闻, 话术, 联系方式, 标书模板, 检测报告, ...]
  - name: 发布日期
    type: date
  - name: 厂商
    type: string
    optional: true
  - name: 产品名
    type: string
    optional: true
  - name: 文档状态
    type: enum
    values: [正式, 草稿]
```
- 长风知识整理系统**入库端**:写 metadata 按此 schema 校验
- 下游系统**检索端**:按此 schema 字段做 Dify metadata filter

#### `agent_kb_roles.yaml` — Agent 角色映射(放 方案生成系统)
```yaml
ProductMatchAgent: [product, supplier_product]
CaseBenchmarkingAgent: [case]
GeographicPolicyAgent: [policy, govt_dynamic]
PainPointStrategicAgent: [case, script]
ScraperAgent: [news_hot]      # 只查近30天新闻
```
- agent 按角色名 → 从 catalog 解析 dataset_ids
- 新增 agent 只改 yaml,不改代码

### 8.2 共享 KB 客户端

- 在长风知识整理系统抽出 **`minimum_workflow/kb_client.py`**
- 以 招投标系统 的 `KBRetriever` 抽象为蓝本,加上:
  - 统一用 `/datasets/{id}/retrieve`
  - 支持按角色名(role)批量检索
  - 支持 metadata filter 参数
  - 内置 LRU 缓存(按 `(datasets, query)` 哈希,10 分钟)
  - retry / health_check
- 方案生成系统 和 招投标系统 都**依赖这个包**(pip install 或 submodule)

### 8.3 角色边界原则

| 系统 | 能做 | 禁止 |
|---|---|---|
| 长风知识整理系统 | **入库**(manual + auto_news)、chunking、metadata 写入、维护三份合约 | - |
| 智能标书(招投标系统) | 检索(按角色/按 dataset id)、本地政策存储、并列展示 | **禁止**向 Dify 直写文档(避免 chunking 冲突) |
| 方案生成(方案生成系统) | 检索(按 agent 角色)、反向飞轮回填(走审核闸门) | **禁止**直接 httpx、**禁止**全库 `search_all` |
| 100MB 标书 workflow | 外部解析 .docx → 提取结构化需求 → 按 role 检索 | **禁止**直接上传 100MB 到 Dify |

### 8.4 api_key 分发策略

每个下游系统独立一把 Dify dataset api_key:
- `招投标系统-key-xxx` 给智能标书
- `方案生成系统-key-xxx` 给方案生成
- `chat-bot-key-xxx` 给 Dify 内聊天机器人
- `tender-workflow-key-xxx` 给 100MB 标书 workflow

Dify 日志可追溯 + 可分别限流/撤销。

### 8.5 库级 ingest 控制

基于用户 §七 澄清(**所有 `自动--` 开头的库是 API 新闻自动抓取,其他都是人工确认**):

- **C 层(资讯)= auto_news**:强制走**生命周期管理**(30 天过期 / 归档)
- **A/B 层 = manual**:**没有自动过期**,由人工审核进出
- 新闻库和人工库**物理隔离**,避免 auto ingest 污染业务库

### 13.2 Dify metadata filter 验证结果 · ❌ 不生效

验证脚本:`D:\changfeng\长风知识整理系统\_dify_metadata_filter_probe.py`
结果文件:`D:\changfeng\长风知识整理系统\_dify_metadata_filter_probe.json`

| 场景 | 过滤条件 | 期望 | 实际 |
|---|---|---|---|
| 1 | 无 | 10 | 10 |
| 2 | `文档分类 contains 检测报告` | ≤ 5 | 10(不过滤) |
| 3 | `文档分类 is 产品参数确认函-整份` | 1 | 10(不过滤) |
| 4 | `生成时间 after 2026-04-01` | <10 | 10(不过滤) |

**结论**:Dify 内网版本**静默忽略** `metadata_filtering_conditions` 参数(HTTP 200,但过滤未执行)。

**附带发现**:`dify_import_service.py:ensure_metadata_fields` 所有字段注册为 `string`,即使 filter 能用,日期比较也无法按 date 执行(需要 type 升级)。

### 13.3 推荐策略:**方案 C 库级物理隔离**

放弃依赖 Dify metadata filter。改为在**入库端**做物理隔离:

- 拆 `自动--公众号` → `自动--公众号--近7天` + `自动--公众号--归档`
- 拆 `自动--新闻日报` → `自动--新闻日报--近90天` + `自动--新闻日报--归档`
- 写**归档脚本**:每日把超期文档从"近期库"move 到"归档库"
- **方案生成系统 的 `news_hot` role 只挂"近期库"**,过滤天然生效(不靠 Dify filter)
- 归档库默认不挂载,需要历史回溯时手动挂

**这件事阻塞点**:归档脚本归属 = 新闻入库系统(同上,待澄清)
